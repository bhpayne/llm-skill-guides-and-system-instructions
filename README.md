# Prompt Engineering

See also
* <https://physicsderivationgraph.blogspot.com/2023/12/best-practices-for-prompts-for-large.html>
* <https://physicsderivationgraph.blogspot.com/2025/06/best-practices-for-use-of-llms.html>

For Physics Derivation Graph: <https://allofphysics.com/blog/page/2026/03/Context%20for%20a%20Prompt%20for%20LLMs>

Prompts without context will have low-quality responses. Provide as much detail as you can. Use domain-specific language. Relevant jargon guides the LLM to a more relevant topic area.

Break large, complicated tasks into smaller, more manageable, and discrete steps that an LLM can handle.<BR>
First Plan: draft a design doc. Then ask "If you were a senior engineer, what are the flaws in this plan?". Then enact the plan by writing software. Then test/evaluate results.

Instead of asking for an entire website or API at once, generate components or single endpoints and build incrementally.

Record which prompts were used to produce what output.

While providing examples can help the LLM with context for what you're seeking, be wary that the examples do not bias the answer towards a certain outcome.

When soliciting feedback from an LLM on content, specify what role or character the LLM should use for the analysis. Should the LLM assess the problem as an experienced professor? Or a sharp commentator on Hacker News?

Specificity matters: Instead of “Summarize this document,” use “Write a 200-word summary focusing on the three main arguments and include a key takeaway for each.”

Specify the tone, style, format, and audience to ensure the output aligns with your needs.

Account for the limited amount of output. Rather than asking for a 10 page report, ask for an outline and then, in separate prompts, fill in the sections.

Iteratively improve your prompts by starting a new session as you learn what guidance the LLM needs to produce relevant output

The more mainstream a topic, the more generic the information an LLM will provide.<BR>
The more mainstream a topic, the more less likely a hallucination is since there's more training data.

LLMs can't reliably do math. LLMs have knowledge cut-off dates.

Ask the LLM to help craft a prompt. For example, "Help me write a detailed and specific prompt to generate a blog post about the benefits of AI in education.”

LLMs are excellent for tasks like generating boilerplate code, summarizing text, explaining complex concepts, and assisting with writing. Ask for advice and brainstorming or feedback on tone and style rather than specific facts. The LLM is not a database of facts.

# Judging output
If you are not able to evaluate the correctness of an answer, do not trust the output of an LLM. Check every claim, every number, every name, every citation.  

LLM code-as-output needs testing and debugging. This should be enacted at each stage: run your app and tests after generating code for each small component to ensure it functions correctly before proceeding.

Don't assume the response of an LLM is self-consistent.

LLMs write with an authoritative and confident tone, regardless of whether the information is correct or nonsensical. This can be misleading.

Indicate which text was generated by an LLM and, in contrast, which text is from a reliable source.

LLMs can inadvertently perpetuate stereotypes related to gender, race, occupations. LLMs are particularly vulnerable to propagating and reinforcing inaccuracies found within their training data, including scientific misconceptions or outdated information commonly perpetuated online.

The first response can be treated as a draft to be iterated upon.

Know when to terminate a session and start a new prompt with relevant context (rather than correcting an existing direction).

If a conversation becomes cluttered or confusing, start a new chat session to reset the context.

View the interaction as a conversation and be prepared to guide the LLM through the process, similar to coaching a human.
